# DESIGN.md — approval-service

Ключевые решения и компромиссы. Команды запуска и справочник API — в [README.md](README.md).

## Границы сервиса

Сервис владеет только процессом согласования: заявка, решение, аудит, события. Публикации, пользователи и workspace — внешние сущности, представленные строковыми идентификаторами (`sourceId`, `userId`, `workspaceId`); их существование и содержимое сервис не проверяет и не хранит. Контент на согласование не передаётся — только `sourceType`/`sourceId` как ссылка на него. Это сознательная граница: сервис невозможно использовать для утечки контента, а интеграция с соседними сервисами происходит асинхронно через события.

## Модель данных (PostgreSQL, Prisma)

- **`approval_requests`** — заявка: `workspaceId`, `sourceType`, `sourceId`, `title`, `description?`, `status`, `reviewerUserIds[]`, `createdByUserId`, поля решения (`decidedByUserId?`, `decisionComment?`, `decisionReason?`, `decidedAt?`), таймстемпы. Индексы `(workspaceId, createdAt)` и `(workspaceId, status)` под список с фильтром и сортировкой; каждый запрос списка/деталки всегда включает `workspaceId` в `WHERE`.
- **`idempotency_keys`** — уникальность `(workspaceId, userId, key)`, fingerprint запроса и сохранённый ответ (status + body).
- **`audit_log`** — кто (`actorUserId`), что (`action: created|approved|rejected|canceled`), по какой заявке, детали без контента и секретов, когда. Отдаётся в деталке заявки как `history[]`.
- **`outbox_events`** — событие для интеграций; `publishedAt IS NULL` — ещё не опубликовано (индекс по `publishedAt`).

Статусы: `pending → approved | rejected | canceled`; все три правых — финальные, переходы между ними запрещены.

## Доступ и изоляция workspace

Аутентификация — заглушка: guard читает `X-User-Id`, `X-Workspace-Id`, `X-Actions` (список разрешённых действий). Предполагается, что в проде эти заголовки проставляет API-gateway после проверки токена, поэтому формат сознательно совпадает с тем, что обычно кладут в клеймы.

- Нет обязательных заголовков → 401.
- `workspace_id` пути ≠ `X-Workspace-Id` → 403. Все запросы к БД дополнительно фильтруются по `workspaceId`, так что чужая заявка неотличима от несуществующей (404) даже при ошибке в guard.
- Действие ручки отсутствует в `X-Actions` → 403.

Доменные проверки поверх этого: approve/reject — только участник `reviewerUserIds` (403 `NOT_A_REVIEWER`), cancel — только `createdByUserId` (403 `NOT_A_REQUESTER`).

## Идемпотентность и повторы

Каждый POST обязан нести `Idempotency-Key` (UUID; без него 400). Обработка:

1. Считается fingerprint = sha256(method + path + body).
2. Бизнес-операция и запись ключа выполняются **в одной транзакции** с заявкой, аудитом и outbox-событием — либо всё, либо ничего.
3. Повтор с тем же ключом и тем же fingerprint → сохранённый ответ (тот же статус и тело), без побочных эффектов.
4. Тот же ключ, другой fingerprint → 422 `IDEMPOTENCY_KEY_REUSE`.
5. Параллельные повторы: гонку ловит unique constraint `(workspaceId, userId, key)` — проигравшая транзакция откатывается целиком (P2002) и отвечает сохранённым ответом победителя.

Компромисс: **кэшируются только успешные ответы**. Ошибки (4xx) не сохраняются — повтор с тем же ключом пересчитывает запрос заново; это позволяет клиенту исправить, например, права и повторить, но означает, что «замороженного» ответа для неуспехов нет. Ключи не имеют TTL/ретеншна — в проде понадобится фоновая очистка старых записей.

Гонки решений: переход статуса выполняется через `updateMany({ where: { id, workspaceId, status: 'pending' } })` внутри транзакции; `count === 0` означает, что заявки нет (404) или она уже финальная (409 после перечитывания). Это исключает двойное решение без `SELECT ... FOR UPDATE`.

## Аудит

Каждое успешное изменение (create/approve/reject/cancel) пишет запись в `audit_log` в той же транзакции, что и само изменение, — аудит не может «отстать» или потеряться. В `details` попадают только доменные поля (комментарий/причина решения, идентификаторы), не контент и не заголовки.

## События и интеграции (outbox)

Публикация напрямую в брокер из обработчика запроса ненадёжна (двухфазной записи «БД + брокер» нет), поэтому используется transactional outbox: событие `approval_request.created|approved|rejected|canceled` пишется в `outbox_events` в транзакции с изменением. Payload собирается только из идентификаторов и статуса — `{ eventId, occurredAt, workspaceId, requestId, sourceType, sourceId, status, actorUserId }`; ни title, ни description, ни комментариев в событиях нет.

Фоновый publisher раз в ~5 секунд выбирает записи с `publishedAt IS NULL`, «публикует» их (в текущей реализации — структурированный лог в stdout) и проставляет `publishedAt`.

Путь к реальному брокеру (Kafka/RabbitMQ): заменяется только шаг «публикации» внутри publisher — вместо лога `producer.send(...)`, `publishedAt` проставляется после подтверждения брокера. Семантика — at-least-once (краш между отправкой и записью `publishedAt` приведёт к повтору), поэтому консюмеры должны быть идемпотентными по `eventId`. Для строгого порядка в партиции ключом сообщения станет `requestId`. При горизонтальном масштабировании publisher-у понадобится `FOR UPDATE SKIP LOCKED` или единственный экземпляр (leader election).

## Что не попадает в ответы, логи и события

- HTTP-логи (pino) редактируют `x-user-id`, `x-workspace-id`, `x-actions`, `authorization`, `cookie`.
- В публичных ответах только поля модели заявки и `history[]`; внутренних служебных таблиц и сырых payload там нет.
- События — только идентификаторы и статус (см. выше). Секретов, токенов, email, storage keys и signed URL в системе нет вовсе — они не принимаются ни в одном поле API.

## Известные компромиссы

- **Auth-заглушка**: заголовкам доверяем без подписи; в проде — проверка JWT на gateway и/или подписанные заголовки. Формат контекста (user, workspace, actions) при этом не меняется.
- **Publisher пишет в лог**, а не в брокер, и работает в единственном экземпляре процесса API (setInterval). При нескольких репликах API события будут публиковаться конкурентно — нужен SKIP LOCKED (см. выше).
- **Offset-пагинация**: проста и достаточна для демо; на больших объёмах и живой ленте корректнее keyset-пагинация по `(createdAt, id)`.
- **`reviewerUserIds` не проверяются на существование** — пользователи внешние; кривой id означает лишь, что такой ревьюер никогда не согласует.
- **Идемпотентность кэширует только успехи**; ошибки пересчитываются (см. раздел про идемпотентность).
- **Один голос решает**: заявка согласуется первым approve любого ревьюера; кворум/мульти-этапность (например, «все ревьюеры») — вне объёма задания, модель позволяет добавить это полем политики на заявке.
- **Миграции на старте контейнера** (`prisma migrate deploy` перед запуском): удобно для compose-стека из одной реплики; при нескольких репликах/CI миграции выносятся в отдельный шаг деплоя.
- **Идемпотентные ключи не чистятся** — нужен ретеншн-джоб в проде.
